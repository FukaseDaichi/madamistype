# マダミスタイプ診断 タイプ設計仕様書

## 1. 文書目的

本書は [specification.md](./specification.md) の「5. タイプ設計」を具体化し、正式なタイプコードとタイプ名称を定義するための補助仕様書である。

本書で定義する内容は以下の通りとする。

- 4軸16タイプのタイプコード規則
- 16タイプの正式名称
- タイプ名称の命名方針
- タイプマスタに登録する初期値

## 2. 設計方針

### 2.1 基本方針

- タイプコードは4文字固定とし、1文字ごとに1つの診断軸へ対応させる
- コード順は診断ロジックと同じく「発言/観察 → 事実/推理 → 論理/感情 → 計画/即興」の順に固定する
- タイプ名称は性格ラベルではなく、マーダーミステリー卓上での立ち回りを想起できる役割名とする
- 名称は拡散しやすさを重視しつつ、ユーザーを下げる印象の語を避ける
- 既存のMBTI名称とは重ならない独自名称とする

### 2.2 本書の位置づけ

- 本書の定義を、初期リリース時点の正式なタイプコード・タイプ名称とする
- 将来ブランド調整で表示名を見直す場合でも、タイプコードは原則として不変とする
- [specification.md](./specification.md) 5.4 の命名イメージは、本書の正式定義で置き換える

## 3. タイプコード設計

### 3.1 コード生成規則

タイプコードは4文字の英大文字で表現する。各文字は以下の診断軸に対応する。

1. 1文字目: 発言型 / 観察型
2. 2文字目: 事実重視 / 推理重視
3. 3文字目: 論理派 / 感情派
4. 4文字目: 計画型 / 即興型

例:

- `TFLP` = 発言型 / 事実重視 / 論理派 / 計画型
- `OREI` = 観察型 / 推理重視 / 感情派 / 即興型

### 3.2 軸と文字の対応表

| 診断軸 | 優勢側 | コード | 対立側 | コード |
| --- | --- | --- | --- | --- |
| 第1軸 | 発言型 | `T` = Talk | 観察型 | `O` = Observe |
| 第2軸 | 事実重視 | `F` = Fact | 推理重視 | `R` = Reasoning |
| 第3軸 | 論理派 | `L` = Logic | 感情派 | `E` = Emotion |
| 第4軸 | 計画型 | `P` = Plan | 即興型 | `I` = Improvisation |

### 3.3 実装ルール

- DB上の主キー相当は `typeCode` を利用する
- UIでは `typeCode` 単体ではなく、タイプ名称とセットで表示する
- SNSシェア文面では `タイプ名称（typeCode）` の形式を基本とする
- 将来的にコピーや説明文を調整しても、`typeCode` は変更しない

## 4. タイプ名称の命名方針

### 4.1 命名ルール

- 名称は「その人が卓でどう動くか」が伝わる職能名・役割名にする
- 名称だけで何となく強みがイメージできるようにする
- 推理ゲーム文脈に合う語彙を使い、過剰に中二的・攻撃的な表現は避ける
- 事実重視タイプには「論証」「証言」「痕跡」などの現場語彙を使う
- 推理重視タイプには「仮説」「裏筋」「物語」「心理」などの読解語彙を使う
- 発言型タイプには場を動かす印象の語を、観察型タイプには拾う・読む・設計する印象の語を優先する

### 4.2 名称構造の考え方

- 前半語: 何を頼りに盤面を見るか
- 後半語: 卓でどのような役回りを担いやすいか

例:

- `論証司令塔` = 証拠と論理を軸に、会話全体を主導するタイプ
- `余白読解者` = 発言されていない感情や含みを読み取るタイプ

## 5. 16タイプ正式定義

| コード | 軸構成 | タイプ名称 | 一言キャッチコピー | 想定プレイ像 |
| --- | --- | --- | --- | --- |
| `TFLP` | 発言 / 事実 / 論理 / 計画 | 論証司令塔 | 証拠を並べ、議論を勝ち筋へ整える。 | 証拠ベースで会話を主導する整理役 |
| `TFLI` | 発言 / 事実 / 論理 / 即興 | 切り込み検証官 | その場の矛盾へ素早く切り込む。 | 反応速度で盤面を動かす検証役 |
| `TFEP` | 発言 / 事実 / 感情 / 計画 | 証言演出家 | 事実を押さえつつ、本音の出る場を作る。 | 証言の引き出し方が上手い進行役 |
| `TFEI` | 発言 / 事実 / 感情 / 即興 | 空気点火役 | 空気を読み、ひと言で卓の温度を変える。 | 場の熱量を上げて情報を回す触媒役 |
| `TRLP` | 発言 / 推理 / 論理 / 計画 | 仮説参謀 | 筋のいい仮説で盤面の地図を描く。 | 複数情報を束ねて読み筋を示す参謀役 |
| `TRLI` | 発言 / 推理 / 論理 / 即興 | 逆転仕掛け人 | 新しい可能性で議論をひっくり返す。 | 発想の切れ味で局面を動かす転換役 |
| `TREP` | 発言 / 推理 / 感情 / 計画 | 感情脚本家 | 心の流れから物語の筋道を組み立てる。 | 動機と関係性を整理して話を導く構成役 |
| `TREI` | 発言 / 推理 / 感情 / 即興 | 物語加速役 | 人の揺れを捉え、ドラマを前へ転がす。 | 感情の変化を起点に展開を進める推進役 |
| `OFLP` | 観察 / 事実 / 論理 / 計画 | 静観分析官 | 多くを語らず、確かな材料で絞り込む。 | 静かに情報を整理して精度を上げる分析役 |
| `OFLI` | 観察 / 事実 / 論理 / 即興 | 痕跡追跡者 | 小さな違和感から真相の糸をたどる。 | 断片的手掛かりを拾い続ける追跡役 |
| `OFEP` | 観察 / 事実 / 感情 / 計画 | 気配記録官 | 言葉の裏にある温度差を静かに残す。 | 表情や空気感を丁寧に蓄積する観測役 |
| `OFEI` | 観察 / 事実 / 感情 / 即興 | 聞き込み遊撃手 | 柔らかく入り込み、本音と手掛かりを拾う。 | しなやかな対話で情報を回収する収集役 |
| `ORLP` | 観察 / 推理 / 論理 / 計画 | 裏筋設計士 | 見えない前提を整理し、真相への設計図を引く。 | 表に出ない構造を組み立てる設計役 |
| `ORLI` | 観察 / 推理 / 論理 / 即興 | 深読み潜行者 | 表に出ず潜り、急所の読みで局面を変える。 | 目立たず核心へ届く深読みに強い奇襲役 |
| `OREP` | 観察 / 推理 / 感情 / 計画 | 心理考察家 | 感情の連なりから人物像と真意を見抜く。 | 人間関係の文脈から全体像を描く読解役 |
| `OREI` | 観察 / 推理 / 感情 / 即興 | 余白読解者 | 沈黙や間に宿る本音を、その場でつかむ。 | 言外の含みを拾って解像度を上げる洞察役 |

## 6. タイプマスタ初期定義

初期リリースのタイプマスタでは、少なくとも以下の項目を保持する。

| 項目名 | 必須 | 内容 |
| --- | --- | --- |
| `typeCode` | 必須 | 本書で定義した4文字コード |
| `typeName` | 必須 | 本書で定義した正式タイプ名称 |
| `tagline` | 必須 | 一言キャッチコピー |
| `summary` | 必須 | 想定プレイ像をもとにした簡易説明 |
| `axis1` | 必須 | 発言型 or 観察型 |
| `axis2` | 必須 | 事実重視 or 推理重視 |
| `axis3` | 必須 | 論理派 or 感情派 |
| `axis4` | 必須 | 計画型 or 即興型 |

## 7. 表示・運用ガイド

### 7.1 結果画面

- 結果画面の最上部は `タイプ名称` を主表示とする
- 補助表示として `typeCode` と `4軸バランス` を併記する
- 一言キャッチコピーはファーストビューに含める

推奨表示例:

```text
論証司令塔
TFLP | 発言型・事実重視・論理派・計画型
証拠を並べ、議論を勝ち筋へ整える。
```

### 7.2 SNSシェア

- シェア文面では名称優先、コード補助とする
- 例: `あなたのマダミスタイプは「論証司令塔（TFLP）」でした`

### 7.3 将来改修時のルール

- キャッチコピーや詳細説明の改修は可能
- タイプ名称の微修正は可能だが、コードとの意味対応を崩さない
- 軸定義を変更する場合は、本書の全面改訂を前提とする
