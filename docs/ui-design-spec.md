# マダミスタイプ診断 UIデザインルール仕様書

## 1. 文書目的

本書は、マダミスタイプ診断のUIデザイン方針を定義するための仕様書である。

本書で定義する内容は以下の通りとする。

- ブランド体験の核となるビジュアルコンセプト
- 色、文字、余白、装飾モチーフのルール
- トップ、設問、結果画面における情報設計
- アクセシビリティとパフォーマンスの基準
- 実装時に参照するデザイントークンと画面構成方針
- OGP画像とSNSシェア時の見せ方

本書は [specification.md](./specification.md) の「4. コンセプト定義」「8. 主要画面詳細」「9. 非機能要件」「11. 文言・表現方針」を、UI観点で具体化する補助仕様書である。

参照文書:

- [specification.md](./specification.md)
- [type-design-spec.md](./type-design-spec.md)
- [tech-stack-spec.md](./tech-stack-spec.md)

## 2. デザインコンセプト

### 2.1 中核コンセプト

本サービスのビジュアルコンセプトは、**「Romantic Case File」** とする。

- アイボリーの紙面に、淡いローズとティールのアクセントを重ねた柔らかい世界観
- 情報は「カード」ではなく「記録物」「証拠」「ファイル」として見せる
- 甘い恋愛診断の雰囲気ではなく、静かな知的没入感と紙面の温かみを両立する
- トップページでは「ケースファイル封筒」モチーフを維持しつつ、ライトトーンで展開する

### 2.2 目指す体験

UIは、以下の感情を順に生むことを目標とする。

1. 「自分のプレイ傾向が記録される」期待感
2. 「事件資料をめくるような」集中感
3. 「この結果は自分らしい」と感じる納得感
4. 「この見た目ごと共有したい」と思える所有感

### 2.3 主要モチーフ

- ケースファイル
- 機密スタンプ
- タイプライター文字
- 手書き注記
- マスキングテープ
- 用紙の罫線、余白、経年感
- アイボリー紙面に淡い色彩で浮かぶ柔らかなコントラスト

### 2.4 キーワード

- 記録
- 捜査資料
- 静かな知的さ
- 紙の手触り
- 読ませる余白
- アイボリーとローズ
- ティールのアクセント

## 3. 対象ユーザーと利用文脈

### 3.1 想定ユーザー

- マーダーミステリーをプレイしたことがある、または興味を持っているユーザー
- 性別を限定せず、物語性、推理、診断コンテンツを楽しめる層
- SNS共有だけでなく、結果ページ自体をじっくり読む層

### 3.2 デバイス前提

- 診断フローはスマートフォン利用を主対象に最適化する
- 一方でトップページは、PC幅でも世界観が立つように横幅を積極的に使う
- モバイルでは1カラムに自然に折りたたみ、PCでは非対称グリッドを許容する

### 3.3 利用文脈

- SNSや共有URLからトップページに到達する
- 数十秒で世界観を理解し、診断開始へ進む
- 診断中は装飾より読みやすさと操作の確実さを優先する
- 結果到達後は、タイプ名、要約、ビジュアル、共有導線をすぐに理解できるようにする

## 4. ブランドトーン

### 4.1 ビジュアルトーン

ビジュアルトーンは **「Romantic Case File」** とする。

- Romantic は甘さではなく、感情の機微や余韻を指す
- Case File は紙の記録物としての知性と整理感を指す
- ライトトーンのアイボリー背景で、紙面の温かみと知的な整理感を両立する

### 4.2 禁止する方向性

- 白背景に淡色カードを並べるだけのSaaS風UI
- ピンク主体の恋愛診断風UI
- 紫主体の占い風UI
- グラスモーフィズム、ネオン、発光表現に依存した演出
- 無意味なミニカードを大量に並べる構成
- 装飾のためだけに複雑化したタイポグラフィ

### 4.3 情報設計の原則

- 1画面に主役は1つだけ置く
- 主役の次に読む情報は2段階までに絞る
- 装飾は階層強調のためだけに使う
- 紙面感を出しても、読みやすさを損なってはならない
- 「雰囲気」より先に「理解できること」を成立させる

### 5.2 運用ルール

- 画面全体の印象は、アイボリー背景にローズとティールのアクセントを散りばめる
- `Rose 600` は主要CTAに限定し、使いすぎない
- `Teal 700` はリンクやアイキャッチラベルに使い、ブランドとの差別化を保つ
- 色だけで意味を伝えず、ラベル、余白、位置でも意味を伝える
- 背景にはローズとティールの放射グラデーションを薄く重ね、奥行きを出す

### 5.3 推奨コントラスト

- アイボリー背景上の本文は `Plum 900` を基準とする
- 12〜14pxのメタ情報でも 4.5:1 を満たす
- ボタン、入力欄、リンクは背景と 3:1 以上のコントラストを確保する

### 6.3 共通スタイル

ボタン、パネル、フォーム要素などの共通スタイルは `app/globals.css` にユーティリティクラスとして定義されている。

- `.surface-panel`: カード面。角丸、半透明背景、内側二重ボーダー

- `.progress-track` / `.progress-fill`: 進捗バー

## 7. タイポグラフィ

### 7.1 役割定義

- **Display**: 見出し、スタンプ、英字ラベル。力強く、証拠タイトルのように見せる
- **Typewriter**: メタ情報、注釈、補足文。記録感を出す
- **Serif**: 日本語の見出し、タイプ名、本文の要点。物語性を支える
- **Note**: 手書き注記やキャッチの一部。使いすぎない
- **Sans**: フォームや設問の可読性が最優先の場面のみ許容する

### 7.2 推奨フォント

全体レイアウト（`app/layout.tsx`）:

- Body: `Noto Sans JP`（`--font-body-family`）
- Display: `Shippori Mincho B1`（`--font-display-family`）

トップページ（`app/page.tsx`）では追加で以下を読み込む:

- Display: `Bebas Neue`（`--rcf-font-display`）
- Typewriter: `Special Elite`（`--rcf-font-typewriter`）
- Serif: `Noto Serif JP`（`--rcf-font-serif`）
- Note: `Caveat`（`--rcf-font-note`）

### 7.3 運用ルール

- 1画面内で主役フォントは最大2系統までに抑える
- 手書き風フォントは注記専用とし、本文には使わない
- タイプライター風フォントは長文本文に多用しない
- 入力欄や設問本文は、演出より可読性を優先する

### 7.4 サイズ基準

| 用途             | 推奨サイズ | 行間     | 備考          |
| ---------------- | ---------- | -------- | ------------- |
| ヒーロータイトル | 48〜96px   | 0.9〜1.0 | Display       |
| セクション見出し | 28〜44px   | 1.0〜1.2 | Display       |
| タイプ名         | 24〜36px   | 1.2〜1.3 | Serif         |
| 本文             | 15〜17px   | 1.7〜1.9 | Serif or Sans |
| メタ情報         | 11〜13px   | 1.6〜1.8 | Typewriter    |
| 注記             | 14〜18px   | 1.3〜1.5 | Note          |

## 8. レイアウト原則

### 8.1 ページごとの重心

- トップページ: PCでは非対称2カラム、モバイルでは1カラム
- 診断フロー: 常に1カラム中心、入力と進行を優先
- 結果ページ: 1カラムを基本とし、PCのみ補助情報を横に逃がす

### 8.2 幅の扱い

- トップページは最大 `1400px` まで使ってよい
- 診断画面は読みやすさ優先で `640px` 前後を基準にする
- 結果画面は `720px` 前後を基準にしつつ、補助情報だけを横展開してよい

### 8.3 グリッド方針

- トップページだけは「意図のある非対称」を許容する
- ただし主役の封筒面、補助の写真面、入力面の3レイヤー以上に増やさない
- 下層ページでの多カラム化は禁止する

### 8.4 装飾ルール

- 紙テクスチャ、赤いマージン線、テープ、スタンプは主役周辺に限定する
- 装飾が可読性を下げる場合は削る
- 1セクションに異なる装飾モチーフを重ねすぎない
- 画像が無い場合でも、プレースホルダーだけで世界観が崩れないようにする

## 9. 画面別ルール

### 9.1 トップページ

#### 主役

- ケースファイル封筒風のヒーロー面
- 診断開始導線

#### 必須構成

1. マストヘッド
2. ケース番号、機密スタンプ、キャッチコピー
3. 32問 / 4ページ / 16タイプ / 4軸 の要約
4. 診断開始CTA
5. 診断で分かること
6. 注目タイプのプレビュー
7. 診断開始フォーム
8. 注目4タイプ
9. 全16タイプ導線

#### ルール

- ヒーローは「説明カード」ではなく「封筒から取り出した資料」に見せる
- サイドバーは補助情報に徹し、主役を奪わない
- フォームはヒーロー直下に置き、CTA後の自然な次の行動にする
- PCでは幅を大胆に使うが、モバイルでは順序を壊さず1カラムにする

### 9.2 診断画面

#### 主役

- 設問と回答操作

#### ルール

- 設問画面では装飾量を下げ、UI密度を整える
- 背景演出は弱め、入力欄と選択肢の視認性を優先する
- 進捗、設問番号、選択状態を明確に見せる
- 未回答エラーは色だけでなく文言と位置でも示す

### 9.3 結果画面

#### 主役

- ユーザー名
- タイプ名
- 一言キャッチコピー
- タイプ別ビジュアル

#### ルール

- 結果ページは「個別の調書」を読む感覚に寄せる
- 最初の画面でタイプ名、要約、共有導線まで理解できる構成にする
- 強み、注意点、向いている役回りは紙面セクションとして区切る
- 4軸表示は補助情報であり、主役にはしない

## 10. OGP画像とSNSシェア

### 10.1 OGP画像方針

- 1200 x 630px を基本とする
- LINEの正方形クロップを考慮し、中央 630 x 630px に主情報を寄せる
- 背景は黒、紙面、タイプ別アクセント色の3層で構成する
- タイプ名、タイプビジュアル、短いコピーだけに絞る
- 文字量を詰め込みすぎない

### 10.2 シェア体験

- 共有時に「診断結果のスクリーンショット感」が出ることを優先する
- コピーしたくなるカード面を作り、その延長としてOGPを整える
- X、LINE、リンクコピーを主要導線とする

## 11. アクセシビリティ

### 11.1 基本原則

- セマンティックHTMLを優先する
- 色だけで意味を伝えない
- フォーカスは常に視認できる
- 装飾はスクリーンリーダーに不要な情報として伝えない

### 11.2 具体ルール

- 通常テキスト 4.5:1、大きいテキスト 3:1 以上
- すべてのタップ対象は 48px 四方以上
- フォーム要素の文字サイズは 16px 未満にしない
- `prefers-reduced-motion: reduce` で演出を止められること
- 画像には意味に応じた `alt` を設定する
- 質問群は `fieldset` + `legend` を使う

## 12. パフォーマンス

### 12.1 基本ルール

- ファーストビューの主役を明確にし、LCP対象を増やしすぎない
- 画像とイラストは必ずサイズを確保する
- アニメーションは `transform` と `opacity` を基本とする
- 黒背景でも重すぎるフィルタ演出を多用しない

### 12.2 目標

- LCP: 2.5秒以内
- INP: 200ms 以内
- CLS: 0.1 以下

## 13. 実装への橋渡し

### 13.1 実装方針

- 全体トークンは `app/globals.css` で管理する
- トップページの強いアートディレクションは `app/page.module.css` で管理する
- フォントは `next/font/google` を使い、外部 `@import` は避ける
- 全体レイアウトでは `Noto Sans JP` と `Shippori Mincho B1` を読み込む
- トップページでは追加で `Bebas Neue`, `Special Elite`, `Noto Serif JP`, `Caveat` を読み込む
- 装飾のために JavaScript を増やさない

### 13.2 実装優先度

1. カラートークンとタイポグラフィ定義
2. トップページの非対称グリッド
3. 診断フローの可読性と操作性
4. 結果画面の主役設計
5. OGP画像の統一感
6. アニメーションと細部演出

### 13.3 品質チェック

- [ ] トップページが 1440px 幅でも間延びしない
- [ ] 900px 未満で自然に1カラムへ落ちる
- [ ] 375px 幅で診断開始まで迷わない
- [ ] 200%ズームでも封筒面とフォームが崩れない
- [ ] キーボードのみで診断開始から完了まで操作できる
- [ ] `prefers-reduced-motion` で大きな演出が止まる
- [ ] 画像未設定時もプレースホルダーだけで世界観が成立する

## 14. 補足

本書は「それっぽいノワール」を作るための資料ではなく、**記録物としての情報設計** と **マーダーミステリーらしい緊張感** を両立させるためのルールである。後続のUI実装では、常に以下の順で判断する。

1. この画面の主役は何か
2. その主役が最も伝わる紙面構成は何か
3. その後で、ケースファイル風の装飾をどこまで足すか
