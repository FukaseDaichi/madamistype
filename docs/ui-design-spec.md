# マダミスタイプ診断 UIデザインルール仕様書 v2.0

## 1. 文書目的

本書は、マダミスタイプ診断のUIデザイン方針を定義するための仕様書である。

本書で定義する内容は以下の通りとする。

- ブランド体験の核となるビジュアルコンセプト

- トップ、設問、結果画面における情報設計
- アクセシビリティとパフォーマンスの基準
- 実装時に参照するデザイントークンと画面構成方針
- OGP画像とSNSシェア時の見せ方

## 2. デザインコンセプト

### 2.1 中核コンセプト

本サービスのビジュアルコンセプトは、**「Noir Case File — 夜の捜査記録」** とする。

- **深夜の捜査室**を想起させる漆黒の背景に、古紙（アイボリー）パネルを差し込む構成
- 情報は「カード」ではなく「証拠書類」「機密ファイル」として見せる
- タイプライター文字・手書き注記・機密スタンプ・罫線紙で紙面の質感を演出する
- 甘さ・占い感・SaaS感をすべて排除し、静かな知的緊張感を維持する

### 2.2 目指す体験

UIは、以下の感情を順に生むことを目標とする。

1. 「これは自分のプレイ記録が作成される」という期待感
2. 「機密文書を開封するような」没入感
3. 「この結果は自分らしい」と感じる納得感
4. 「この見た目ごと共有したい」と思える所有感

### 2.3 主要モチーフ

| モチーフ                                        | 用途                         | 注意                             |
| ----------------------------------------------- | ---------------------------- | -------------------------------- |
| 深夜の捜査室（黒背景）                          | 画面全体の基調               | グラデーションより単色黒を優先   |
| 古紙パネル（アイボリー罫線紙）                  | ヒーロー・ヘッダー情報       | 主役周辺にのみ使用               |
| 機密スタンプ（Confidential / Shared Result 等） | 各ページのモード表示         | 回転・透明度で押印感を出す       |
| タイプライター文字（Special Elite）             | メタ情報・注釈・補足文       | 本文の長文には使わない           |
| 手書き注記（Caveat）                            | キャッチ・サブコピー一部     | 使いすぎない。1面に1箇所まで     |
| 左マージンライン（薄い赤縦線）                  | 古紙パネル内の装飾           | 古紙パネルの外には出さない       |
| 横罫線テクスチャ                                | 古紙パネルの背景             | repeating-linear-gradient で実装 |
| テープ（マスキングテープ風）                    | 写真・アートワーク貼り付け   | ::before 疑似要素で実装          |
| タイプコード分解バッジ                          | タイプ詳細ページのヒーロー下 | 黒背景 + 琥珀文字                |
| 琥珀色の数値（Bebas Neue）                      | メタバッジの強調数値         | amber (#d4820a) 固定             |

### 2.4 キーワード

- 機密記録
- 深夜の捜査室
- 古紙と黒のコントラスト
- タイプライターの打鍵感
- 静かな知的緊張
- 証拠書類としての情報整理

## 3. 対象ユーザーと利用文脈

### 3.1 想定ユーザー

- マーダーミステリーをプレイしたことがある、または興味を持っているユーザー
- 性別を限定せず、物語性・推理・診断コンテンツを楽しめる層
- SNS共有だけでなく、結果ページ自体をじっくり読む層

### 3.2 デバイス前提

- 診断フローはスマートフォン利用を主対象に最適化する
- トップページは PC 幅（最大 1400px）でも世界観が立つよう非対称グリッドを使う
- モバイルでは1カラムに自然に折りたたみ、PCでは非対称グリッドを許容する

### 3.3 利用文脈

- SNSや共有URLからトップページに到達する
- 数十秒で世界観を理解し、診断開始へ進む
- 診断中は装飾より読みやすさと操作の確実さを優先する
- 結果到達後は、タイプ名・要約・ビジュアル・共有導線をすぐに理解できるようにする

## 4. ブランドトーン

### 4.1 ビジュアルトーン

ビジュアルトーンは **「Noir Case File」** とする。

- **Noir** は漆黒の背景・光と影のコントラスト・静かな緊張感を指す
- **Case File** は紙の記録物としての知性と整理感、証拠書類としての権威を指す
- ライトトーンや柔らかな世界観への回帰は行わない

### 4.2 禁止する方向性

- アイボリー背景に淡色カードを並べるライトトーンUI（v1.0 スタイル）
- ピンク主体の恋愛診断風UI
- 紫主体の占い風UI
- グラスモーフィズム、ネオン、発光表現への依存
- 白背景・グレー背景への変更
- Rose / Teal カラーシステム（v1.0）の復活
- 無意味なミニカードの大量配置
- 装飾のためだけに複雑化したタイポグラフィ

### 4.3 情報設計の原則

- 1画面に主役は1つだけ置く
- 主役の次に読む情報は2段階までに絞る
- 装飾は階層強調のためだけに使う
- 古紙パネルは「主役情報の器」として使い、背景装飾には使わない
- 「雰囲気」より先に「理解できること」を成立させる

### 5.2 カラー運用ルール

- **背景は常に `#0d0b08`（Noir）**。グレーや白への変更は不可
- **古紙パネル（`#f0e8d5`）は主役ヒーロー・ヘッダーのみ**に使用し、セクションカードには使わない
- **黒パネル（`rgba(10,8,5,0.88)`）** をセクション情報の器として統一使用する
- `#c0392b`（Red）はCTA・スタンプ・リスト装飾（ダッシュ）に限定し、乱用しない
- `#d4820a`（Amber）は強調数値・タイプコード文字のみ
- 色だけで意味を伝えず、ラベル・余白・位置でも意味を伝える

### 5.3 5段階スケール専用カラー（診断画面）

診断の5段階回答スケールのペンアイコン色は下記を固定とする。

| 値  | 意味             | stroke色                 |
| --- | ---------------- | ------------------------ |
| 5   | とてもそう思う   | `#c0392b`                |
| 4   | ややそう思う     | `#a03020`                |
| 3   | どちらでもない   | `rgba(196,169,107,0.45)` |
| 2   | ややそう思わない | `#1a5a50`                |
| 1   | 全くそう思わない | `#0d4a40`                |

未選択時はすべて `rgba(196,169,107,0.22)` に統一し、選択時にのみ上記色へトランジションさせる。

### 5.4 コントラスト基準

- 古紙パネル上のメインテキスト（`#1a1208` on `#f0e8d5`）: 十分なコントラストを確保
- 黒パネル上の本文（`rgba(240,232,213,0.85)` on `rgba(10,8,5,0.88)`）: 同上
- ボタン・リンクは背景と 3:1 以上のコントラストを確保する

## 7. タイポグラフィ

### 7.1 フォント体系

| 役割       | CSS変数                 | フォント名    | 主な用途                                                          |
| ---------- | ----------------------- | ------------- | ----------------------------------------------------------------- |
| Display    | `--rcf-font-display`    | Bebas Neue    | 大見出し・スタンプ・セクションタイトル・CTAボタン・メタバッジ数値 |
| Typewriter | `--rcf-font-typewriter` | Special Elite | メタ情報・注釈・ファイル番号・リスト本文・セカンダリボタン        |
| Serif      | `--rcf-font-serif`      | Noto Serif JP | 日本語本文・タイプ名・質問テキスト・相性テキスト                  |
| Note       | `--rcf-font-note`       | Caveat        | 手書き注記・サブコピー（1面に1箇所まで）                          |

全体レイアウト（`app/layout.tsx`）では追加で以下を読み込む:

- Body: `Noto Sans JP`
- Display: `Shippori Mincho B1`（タイプ詳細以外の汎用日本語見出し）

### 7.2 フォント読み込み

フォントは `next/font/google` を使い、外部 `@import` は避ける。

```tsx
// トップページ・詳細ページ共通で使用するフォント
const displayFont    = Bebas_Neue({ variable: '--rcf-font-display',     weight: '400', ... });
const typewriterFont = Special_Elite({ variable: '--rcf-font-typewriter', weight: '400', ... });
const serifFont      = Noto_Serif_JP({ variable: '--rcf-font-serif',      weight: ['400','700'], ... });
const noteFont       = Caveat({ variable: '--rcf-font-note',              weight: ['400','700'], ... });
```

### 7.3 運用ルール

- 1画面内で主役フォントは最大2系統まで
- 手書き風（Caveat）は注記専用。本文・リストには使わない
- タイプライター（Special Elite）は英字・メタ情報専用。日本語長文は Noto Serif JP を使う
- 質問本文・入力欄は Noto Serif JP または Noto Sans JP を優先し、可読性を確保する
- 古紙パネル上の見出しは Bebas Neue + Ink（`#1a1208`）
- 黒パネル上の見出しは Bebas Neue + `rgba(240,232,213,0.9)`

### 7.4 サイズ基準

| 用途               | サイズ               | 行間     | フォント      |
| ------------------ | -------------------- | -------- | ------------- |
| ページ主見出し     | 2.4rem〜4rem         | 0.92     | Bebas Neue    |
| セクションタイトル | 1.4rem〜2rem         | 1.0      | Bebas Neue    |
| タイプコード文字   | 3.2rem（アイコン内） | 1.0      | Bebas Neue    |
| 質問本文           | 0.82rem（11px相当）  | 1.85     | Noto Serif JP |
| リスト本文         | 0.8rem               | 1.8      | Special Elite |
| 詳細テキスト       | 0.8〜0.82rem         | 2.0〜2.1 | Special Elite |
| メタ情報           | 0.62〜0.72rem        | 1.6      | Special Elite |
| 手書き注記         | 0.9〜1.1rem          | 1.4      | Caveat        |
| ファイル番号       | 0.68〜0.72rem        | —        | Special Elite |

## 8. レイアウト原則

### 8.1 ページごとの重心

| ページ       | 基本構成                                                 | 最大幅 |
| ------------ | -------------------------------------------------------- | ------ |
| トップページ | PC: 非対称2カラム + 下段フォームゾーン / Mobile: 1カラム | 1400px |
| 診断フロー   | 常に1カラム中心                                          | 760px  |
| タイプ詳細   | 1カラム基本                                              | 960px  |
| 共有結果     | 1カラム基本                                              | 900px  |

### 8.2 グリッド方針

- **トップページのみ** 非対称グリッド（`grid-template-columns: 1fr 440px`）を許容する
- 古紙ヒーロー面・サイドバー・フォームゾーンの3エリア以上に増やさない
- 下層ページでの多カラム化は **強み/注意 の2カラム** に限定する

### 8.3 装飾ルール

- **古紙パネル**（罫線テクスチャ・左マージンライン・テープ）は主役のヒーローエリアに限定する
- **スタンプ**（Confidential / Shared Result / On Record / Public File）は各ページに1個のみ
- **テープ疑似要素**（`::before` による `rgba(210,190,140,0.5)` の横帯）はアートワーク貼り付けにのみ使用
- 1セクションに異なる装飾モチーフを重ねない
- 装飾が可読性を下げる場合は即座に削る

### 8.4 アニメーション方針

- `transform` と `opacity` を基本とし、`filter` や `backdrop-filter` の多用は禁止
- フェードイン（`fadeUp`）は古紙パネル・質問パネルのページ初回表示のみに使用
- 診断スケールのペンアイコンは `transform: scale` + `stroke` トランジションのみ
- `prefers-reduced-motion: reduce` で全アニメーションを停止できること

## 9. 画面別ルール

### 9.1 トップページ（`app/page.tsx`）

#### 主役

- ケースファイル封筒風の古紙ヒーローパネル（左カラム）
- 診断開始導線

#### 必須構成（上から順）

1. マストヘッド（ロゴ + ナビゲーション）
2. 古紙ヒーローパネル（ケース番号・スタンプ・キャッチ・統計バッジ・CTA・Featureカード）
3. サイドバー（Preview Typeカード + LINE スタンプ）
4. 診断フォームゾーン（黒背景 + `Start Diagnosis` ラベル + フォーム）
5. Featured Types（4カラムグリッド）
6. 全16タイプ一覧（chipグリッド）
7. フッター

#### ルール

- ヒーローは「説明カード」ではなく「封筒から取り出した捜査資料」に見せる
- サイドバーは右カラムに収め、主役を奪わない（幅上限 440px）
- フォームゾーンは黒背景パネルで主役カードと区別し、CTAの自然な着地点にする
- PCでは幅（最大1400px）を大胆に使う。モバイルでは順序を壊さず1カラムに折りたたむ
- Featured Types の4枚カードは上部カラーボーダー（赤/琥珀/緑/青）でタイプを識別する

### 9.2 診断フロー（`app/diagnosis/`）

#### 主役

- 質問テキストと5段階回答スケール

#### 画面構成

1. ヘッダーパネル（古紙）: ファイル番号・ユーザー名・メタバッジ・プログレスバー
2. 質問パネル（黒）: セクションラベル・質問カード群・ナビゲーション

#### ルール

- ヘッダーパネルは古紙モチーフを適用し、スタンプは `On Record`
- 質問パネルは黒パネルで統一し、余分な装飾を加えない
- 各質問カードは `border-bottom` のみで区切る（囲み枠は使わない）
- 回答済みカードには `border-left: 2px solid rgba(196,169,107,0.3)` のみで表現する

#### 5段階スケール UIルール

```
[ 大赤ペン ] [ 小赤ペン ] [ 極小灰ペン ] [ 小緑ペン ] [ 大緑ペン ]
  そう思う                                            そう思わない
```

- アイコンは区切り線・背景なしで一列に並べる
- 「そう思う」「そう思わない」ラベルは**スケール行の下**に左右配置する（各ボタン内には書かない）
- 未選択時: 全アイコン `rgba(196,169,107,0.22)` で統一表示
- 選択時: 選択アイコンのみ固有色へトランジション（`transition: stroke 0.3s ease`）
- 選択時: アイコンが `transform: scale(1.18)` でバウンスアップ（`cubic-bezier(.34,1.56,.64,1)`）
- 選択時: 他アイコンは `opacity: 0.3` にフェードアウト
- 選択時: ラベル行の中央にインジケーターバーが出現（色・幅で選択位置を表現）
- 選択時: アイコン下に `glow`（blur付きの小さい光点）が出現
- ナビゲーション: 左「← 前の8問」(セカンダリ) / 右「次の8問へ →」(プライマリ赤)

### 9.3 タイプ詳細ページ（`app/types/[typeCode]/`）

#### 主役

- タイプ名・タグライン・アートワーク
- 4軸構成ビジュアル

#### 画面構成

1. マストヘッド（ロゴ + 戻るリンク）
2. 古紙被疑者カード（スタンプ・見出し・手書き注記・アートワーク2カラム・タイプコード分解バッジ・アクション）
3. Type Signature セクション（黒）: 4軸カードグリッド
4. 強み / 注意点（2カラム黒パネル）
5. 詳しい見立て（黒パネル）
6. 向いている立ち回り / 役回り（2カラム黒パネル）
7. 相性の傾向（黒パネル）
8. 共有パネル（黒）
9. フッター

#### 4軸カードのルール

- 各カードは `優勢文字（大）+ 残りの文字（小）` の大小分割表示
  - 例: `T` (大・琥珀) + `alk` (小・薄い琥珀) → `Talk / 発言型`
- 優勢側: 大きな琥珀色 (`#d4820a`)
- 劣勢側: 小さく薄い色 (`rgba(196,169,107,0.15)`)
- VS 区切り線で上下に分割
- 傾向テキストは左赤ボーダー付きの1〜2文に凝縮する
- `優勢` バッジ（赤枠）を右上に配置する

#### タイプコード分解バッジのルール

- アートワーク+タイプ情報の直下・アクションボタンの直上に配置する
- `T · F · L · P` を各1バッジで表示（黒背景 + 琥珀文字）
- バッジ内: 大文字（1.8rem・amber）/ 英単語（0.7rem・sepia-light）/ 日本語（0.72rem・sepia）

## 10. OGP画像とSNSシェア

### 10.1 OGP画像方針

- 1200 × 630px を基本とする
- LINEの正方形クロップを考慮し、中央 630 × 630px に主情報を寄せる
- **背景は黒（`#0d0b08`）**。ライトトーンへの変更は禁止
- タイプ名（Bebas Neue）・タイプビジュアル・短いコピーだけに絞る
- 機密スタンプ風装飾を1点のみ追加可

### 10.2 シェア体験

- 共有時に「捜査記録のスクリーンショット感」が出ることを優先する
- X・LINE・リンクコピーを主要導線とする
- 共有 URL は `/types/[typeCode]/[key]` 形式とし、個人名は URL に含めない

## 11. アクセシビリティ

### 11.1 基本原則

- セマンティックHTMLを優先する（`fieldset` + `legend` / `section` / `header` / `footer`）
- 色だけで意味を伝えない（スケール選択は色 + インジケーターバー + グロウで多重表現）
- フォーカスは常に視認できる
- 装飾要素（スタンプ・テープ・罫線）は `aria-hidden="true"` を付与する

### 11.2 具体ルール

- 通常テキスト 4.5:1、大きいテキスト 3:1 以上のコントラスト比
- すべてのタップ対象は 48px 四方以上（診断スケールの各ボタンを含む）
- フォーム要素・質問テキストの文字サイズは 16px 未満にしない
- `prefers-reduced-motion: reduce` でアニメーション（scale・opacity トランジション）を停止する
- 5段階スケールの各ラジオボタンには `aria-label`（例: `とてもそう思う`）を設定する
- 質問群は `fieldset` + `legend` を使い、`legend` は視覚的非表示（`display: contents`）にしてもよい

## 12. パフォーマンス

### 12.1 基本ルール

- ファーストビューの主役（古紙ヒーローパネル）を明確にし、LCP対象を増やさない
- アートワーク画像は `priority` 属性を付与し先読みする
- アニメーションは `transform` と `opacity` を基本とし、`filter: blur` は診断スケールのグロウにのみ限定使用する
- 背景にはグラデーションより単色を優先し、`backdrop-filter` は使わない

### 12.2 目標

- LCP: 2.5秒以内
- INP: 200ms以内
- CLS: 0.1以下

## 13. 実装への橋渡し

### 13.1 実装方針

- カラートークンは `app/globals.css` の `:root` で一元管理する
- トップページのアートディレクションは `app/page.tsx` の `<style>` タグ（または `page.module.css`）で管理する
- フォントは `next/font/google` を使い、外部 `@import` は避ける
- 全体レイアウトでは `Noto Sans JP` と `Shippori Mincho B1` を読み込む
- トップ・詳細ページでは追加で `Bebas Neue`, `Special Elite`, `Noto Serif JP`, `Caveat` を読み込む
- **装飾のために JavaScript を増やさない**（テープ・罫線・マージンラインは CSS のみ）

### 13.2 実装優先度

1. カラートークンとタイポグラフィ定義（本書 5〜7章）
2. トップページの非対称グリッド（幅 1400px・PC/Mobile 切り替え）
3. 診断フローの5段階スケール UI（アニメーション含む）
4. タイプ詳細の4軸カードとタイプコード分解バッジ
5. 共有結果ページの差分実装
6. OGP画像の統一感

### 13.3 品質チェック

- [ ] トップページが 1440px 幅でも間延びしない
- [ ] 900px 未満で自然に1カラムへ落ちる
- [ ] 375px 幅で診断開始まで迷わない
- [ ] 200% ズームでも古紙パネルとフォームが崩れない
- [ ] キーボードのみで診断開始から完了まで操作できる
- [ ] `prefers-reduced-motion` でスケールアニメーションが停止する
- [ ] 画像未設定時もプレースホルダーだけで世界観が成立する（Bebas Neue コード文字 + 薄い border）
- [ ] 黒背景のまま白・グレーへの「取り消し」が発生していない
