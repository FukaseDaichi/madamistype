# マダミスタイプ診断 診断ロジック仕様書（たたき台）

## 1. 文書目的

本書は [specification.md](./specification.md) の「7. 診断ロジック概要」を具体化し、MVP時点で実装可能な診断ロジックのたたき台を定義するための補助仕様書である。

本書で定義する内容は以下の通りとする。

- 診断ロジックの前提条件
- 質問マスタの設計方針
- 回答値のスコアリング方法
- 4軸の優勢判定方法
- 16タイプへのマッピング方法
- 結果表示に必要な派生値

## 2. 参照文書

- [specification.md](./specification.md)
- [type-design-spec.md](./type-design-spec.md)

## 3. MVP前提

### 3.1 前提条件

- 質問数は全32問とする
- 4軸それぞれに8問ずつ割り当てる
- 回答形式は5段階評価とする
- 1問は原則として1軸の判定にのみ使う
- MVPでは、設問ごとの複数軸加点は行わない
- MVPでは、相性情報は静的表示とし、診断ロジックには含めない

### 3.2 軸の並び順

タイプコードの軸順は [type-design-spec.md](./type-design-spec.md) に合わせ、以下で固定する。

1. 発言型 / 観察型
2. 事実重視 / 推理重視
3. 論理派 / 感情派
4. 計画型 / 即興型

### 3.3 軸の優勢側定義

各軸の正方向を以下の通り定義する。

| 軸 | 正方向 | 負方向 | コード |
| --- | --- | --- | --- |
| 第1軸 | 発言型 | 観察型 | `T` / `O` |
| 第2軸 | 事実重視 | 推理重視 | `F` / `R` |
| 第3軸 | 論理派 | 感情派 | `L` / `E` |
| 第4軸 | 計画型 | 即興型 | `P` / `I` |

## 4. 質問設計方針

### 4.1 質問配分

- 各軸8問とし、合計32問を構成する
- 各軸8問のうち、4問は正方向設問、4問は逆方向設問を推奨する
- 全設問を4ページに分割し、1ページ8問で表示する
- 各ページに4軸が偏りすぎないよう、軸を分散して配置する

### 4.2 質問文の原則

- 1問で複数の解釈が生まれない文にする
- マーダーミステリー中の振る舞いが想像できる文脈に寄せる
- 能力の優劣ではなく、行動傾向を聞く
- 「いつも」「絶対」などの極端な表現は避ける
- 良し悪しが透けて見える誘導的な文言は避ける

### 4.3 設問方向

各設問には `direction` を持たせる。

- `forward`: 正方向に同意が強まる設問
- `reverse`: 負方向に同意が強まる設問

例:

- `forward`: 「議論が止まると、自分から話を切り出すほうだ」
- `reverse`: 「自分から話すより、まず周囲の出方を見たい」

## 5. 質問マスタ仕様

### 5.1 必須項目

MVPの質問マスタには少なくとも以下を持たせる。

| 項目名 | 必須 | 内容 |
| --- | --- | --- |
| `questionId` | 必須 | 質問の一意ID |
| `questionText` | 必須 | 質問文 |
| `axis` | 必須 | `A1` `A2` `A3` `A4` のいずれか |
| `direction` | 必須 | `forward` または `reverse` |
| `displayOrder` | 必須 | 全体の表示順 |
| `pageNo` | 必須 | 表示ページ番号 |
| `weight` | 必須 | MVPでは原則 `1.0` 固定 |
| `isActive` | 必須 | 出題対象フラグ |
| `tieBreakerPriority` | 任意 | 同点時の参照優先度 |

### 5.2 MVP時点の運用ルール

- `weight` は将来拡張用に持つが、MVPでは全問 `1.0` とする
- `axis` は必ず1つだけ設定する
- 無効化した質問は集計対象から除外する
- 本番投入前に、軸ごとの質問数が8問ずつであることを検証する

## 6. 回答値とスコアリング

### 6.1 回答値の数値化

回答値は以下の通り数値化する。

| 回答 | 値 |
| --- | --- |
| とてもそう思う | 5 |
| ややそう思う | 4 |
| どちらでもない | 3 |
| あまりそう思わない | 2 |
| まったくそう思わない | 1 |

### 6.2 軸加点用の変換

集計では、回答値をそのまま合計せず、軸の正負が分かる差分値へ変換する。

- `forward` の場合: `delta = answerValue - 3`
- `reverse` の場合: `delta = 3 - answerValue`

これにより、各設問の寄与値は `-2` から `+2` の範囲になる。

| 回答 | `forward` | `reverse` |
| --- | --- | --- |
| 5 | +2 | -2 |
| 4 | +1 | -1 |
| 3 | 0 | 0 |
| 2 | -1 | +1 |
| 1 | -2 | +2 |

### 6.3 軸スコアの計算式

軸スコアは以下で求める。

```text
axisScore = Σ(delta × weight)
```

MVPでは `weight = 1.0` 固定のため、実質的には各設問の `delta` 合計となる。

各軸8問、各問 `-2` から `+2` のため、軸スコアの範囲は `-16` から `+16` となる。

### 6.4 集計例

第1軸の8問について、`delta` の合計が `+5` の場合:

- 第1軸は正方向優勢
- タイプコード上は `T` を採用する

第2軸の合計が `-3` の場合:

- 第2軸は負方向優勢
- タイプコード上は `R` を採用する

## 7. タイプ決定ロジック

### 7.1 基本判定

各軸の `axisScore` から優勢側を決定する。

- `axisScore > 0`: 正方向
- `axisScore < 0`: 負方向
- `axisScore = 0`: 同点扱い

### 7.2 同点時の暫定ルール

MVPでは、同点時に以下の順で判定する。

1. 該当軸の設問のうち、`tieBreakerPriority` が最も高い設問を参照する
2. その設問の `delta` が正なら正方向、負なら負方向とする
3. その設問が `0` の場合は、次順位の設問を参照する
4. 全設問が `0` の場合は、軸ごとの既定値を採用する

MVPの軸ごとの既定値は以下の通りとする。

- 第1軸: 観察型
- 第2軸: 事実重視
- 第3軸: 論理派
- 第4軸: 計画型

この既定値は暫定であり、プレテスト後に見直す前提とする。

### 7.3 タイプコード変換

4軸の判定結果を、[type-design-spec.md](./type-design-spec.md) のコード規則へそのまま変換する。

例:

- 第1軸 `T`
- 第2軸 `F`
- 第3軸 `L`
- 第4軸 `P`

この場合のタイプコードは `TFLP` となる。

### 7.4 タイプ名称の取得

- 算出したタイプコードをキーとしてタイプマスタを参照する
- タイプ名称、一言キャッチコピー、説明文、強み、注意点、向いている役回りを取得する
- 相性情報も同じタイプマスタから取得する

## 8. 結果表示用の派生値

### 8.1 4軸バランス表示

結果画面の4軸バランス表示には、軸スコアを 0 から 100 の値へ正規化して利用する。

```text
balanceScore = round(((axisScore + 16) / 32) * 100)
```

解釈例:

- `0`: 負方向に大きく寄る
- `50`: 中央
- `100`: 正方向に大きく寄る

### 8.2 表示ラベル

バランス表示には数値だけでなく、軸ラベルも併記する。

例:

- 発言型 68 / 観察型 32
- 事実重視 41 / 推理重視 59

### 8.3 補足指標

将来拡張として、以下の派生値も追加可能とする。

- 軸ごとの判定差分の強さ
- 回答の一貫性指標
- タイプ確信度に近い補助指標

ただし、MVPでは表示しない。

## 9. 相性情報の扱い

### 9.1 基本方針

- 相性情報は診断ロジックで算出しない
- タイプごとに固定文言を持たせ、結果画面に静的表示する
- ユーザーの相手情報や複数人の診断結果を入力して比較する機能はMVP対象外とする

### 9.2 タイプマスタ項目案

相性情報として、タイプマスタに以下のいずれかを持たせる。

- `compatibilitySummary`: 相性傾向の要約文
- `goodWithTypeCodes`: 相性が良い傾向として見せる代表タイプコード一覧
- `goodWithDescription`: なぜ相性が良いかの説明文

MVPでは `compatibilitySummary` のみでも成立する。

## 10. 実装時の検証観点

### 10.1 マスタ検証

- 質問数が32問であること
- 各軸8問ずつであること
- `direction` の偏りが大きすぎないこと
- `displayOrder` と `pageNo` が矛盾しないこと
- タイプマスタが16件揃っていること

### 10.2 計算検証

- 全設問回答時のみ結果計算が走ること
- 同じ回答セットなら必ず同じタイプコードになること
- 同点時ルールが決定的に動作すること
- 4軸バランス表示値が `0` から `100` に収まること

### 10.3 表示検証

- タイプコードとタイプ名称の組み合わせが正しいこと
- 相性情報が動的計算ではなく静的文言から取得されること
- 結果画面で全項目が閲覧できること
